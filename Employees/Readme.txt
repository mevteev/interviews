Вот что у меня получилось - 

Объект Сотрудник имеет свойства: Name, Title, ссылка на руководителя (Manager), 
id - внутренний уникальный идентификатор.

Чтение/сохранение в XML я сделал по простому - через FileReader/FileWriter.

Проблему с долгой записью я решил обойти следующим способом -
Каждый раз при редактировании/удалении/добавлении новые данные сохраняются не в исходный массив  сотрудников, а в буферную таблицу, где указывается содержимое измененного объекта, время изменения, и статус изменения (ADD, EDIT, DELETE)

При каждом изменении объекта мы ставим флаг, что нужно сохранить данные в файл. И, если на текущий мометн не идет процесс записи, запускаем поток, в котором происходит запись содержимого массива сотрудников + содержимое буферной таблицы (берем только записи с временем изменения меньше, чем время старта потока записи), таким образом, если в массиве есть сотрудник с id = X, но в буферной таблице есть запись, что мы сотрудника Х удалили, то в файл сотрудник не попадает.
Аналогично, если в буф.таблице есть новый id, а в массиве нету, новая запись попадет в файл.

Это необходимо чтобы избегать следующей ситуации: если после начала записи файла будет удален к примеру сотрудник в конце списка, а до этого в файл уже попали его "подчиненные", то после завершения записи данные в файле будет некоррекны (поле Manager будет ссылаться на несуществующую запись)

Что не успел сделать:
1. После успешной записи файла нужно содержимое буферной таблицы переместить исходный массив сотрудников
2. Также, не доделал в форме редактирования выбор в комбобоксе корректного подмножества возможных руководителей для редактируемого сотрудника. Сейчас можно выбрать любого сотрудника в качестве руководителя, что приводит к закольцованности орг.структуры предприятия.
Тут необходимо выводить разность множеств [Все сотрудники] - [Подчиненные сотрудника] - [подчиненные подчиненных] - ...
Не успел написать процедуру построения дерева структуры подчиненности...
